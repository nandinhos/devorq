**Introdução e o Novo Workflow**
No vídeo de hoje eu vou mostrar para vocês o workflow completo que eu uso para criar aplicações profissionais com o Claude Code, e esse workflow é bem diferente do Vibe Coding que você conhece. Eu estou falando de criar aplicações profissionais, como é o caso do Epic New, um produto que lançamos e que em um dia bateu os seus primeiros 300 usuários. Absolutamente ninguém acredita que ele foi feito 100% com IA, e se eu tivesse feito ele no Vibe Coding, eu garanto para vocês que teria sido impossível concluir esse projeto, porque uma coisa é fazer um protótipo simples e outra coisa é fazer uma aplicação complexa. O meu workflow é basicamente o seguinte: primeiro, eu começo escrevendo uma 'spec', que é um documento que lista todas as páginas, os componentes e os comportamentos dos usuários na aplicação, o que ajuda a ter clareza sobre o que vou construir e a passar essa clareza para o Claude. Depois, eu quebro essa spec, que é um documento bem grande, em várias tarefinhas pequenininhas. Isso é muito importante para que o Claude não engasgue fazendo uma implementação, porque se você der uma tarefa muito grande, ele vai acabar engasgando no meio do caminho. Para cada tarefinha, eu faço um planejamento buscando na base de código o que posso reutilizar ou importar, além de olhar documentações externas para encontrar padrões comprovados, evitando reinventar a roda. Por fim, a quarta etapa é realmente implementar esse planejamento da tarefinha.

**Os Problemas do "Vibe Coding Vida Louca"**
À primeira vista, esse processo pode parecer um pouco burocrático para quem vem do extremo do "Vibe Coding Vida Louca Total", mas chegamos a ele como um antídoto após descobrirmos alguns padrões problemáticos dessa abordagem. Um erro muito comum é dar uma tarefa muito grande para a IA implementar; isso lota a janela de contexto dela, e queremos manter essa janela o mais limpa possível para aproveitar a maior performance do modelo. Outro problema bem comum é que o código gerado pela IA é muito bagunçado, pois ela tende a complicar o que deveria ser simples, tenta implementar coisas proativamente que você não pediu ou reinventa a roda. A IA também costuma repetir trechos de código que já existem (como recriar um botão já existente) simplesmente porque não se lembra que já os escreveu, tornando a manutenção um pesadelo. 

Muitas vezes as pessoas dizem que "a IA não obedece" ou que o Claude é burro, mas na verdade os usuários focam em explicar *o que* querem fazer, e não o *como* fazer. Se você não disser exatamente quais arquivos a IA deve modificar ou criar, ela vai tentar adivinhar, e muitas vezes mexe no arquivo errado, o que só piora conforme o projeto cresce. Há também o problema do "cobertor de pobre" (arrumar uma coisa e quebrar outra), que ocorre porque a IA junta responsabilidades que deveriam estar isoladas; como ela não modulariza o código, alterar algo afeta outras partes. Por último, existem as gafes de segurança, que geralmente ocorrem porque a IA coloca lógica de negócio ou chaves expostas no front end. Tudo que está no front end pode ser acessado com dois cliques no navegador, então todas as regras de negócio devem ficar no seu back end.

**As Soluções Aplicadas no Novo Workflow**
Para resolver o problema da IA engasgando, ao invés de dar um projetão, você quebra o seu projeto em tarefinhas pequenas e entrega uma por uma para preservar a janela de contexto. Para evitar código bagunçado e repetido, eu peço para a IA rodar uma pesquisa no projeto e identificar trechos que posso importar, além de pesquisar padrões de implementações documentados. Para fazer a IA te obedecer, a melhor prática é dizer exatamente qual arquivo ela tem que criar ou modificar, e o que tem que ser feito em cada um. Para evitar o problema de quebrar uma coisa ao consertar outra, adotamos uma arquitetura organizada por comportamento, onde cada comportamento fica isoladinho na sua respectiva pasta. Já para as gafes de segurança, direcionamos o Claude com a regra "thin client, fat server": o front end não controla nenhuma regra de negócio, apenas captura as intenções do usuário e manda para o back end retornar a resposta, e nunca chumbamos chaves expostas.

**O Passo a Passo na Prática**
Na prática, a primeira coisa que eu faço é usar o comando `/spec` para criar o documento explicando pro Claude o projeto, listando as páginas, componentes e comportamentos. Depois, rodo o comando `break` e arrasto a minha spec; ele pega o documento e quebra em várias tarefas (issues), onde cada página e cada comportamento viram uma tarefinha, começando sempre pelo protótipo (apenas a tela visual). 

Antes de implementar, rodo o comando `plan` na issue desejada; isso faz a IA pesquisar na base de código e na internet para planejar a execução sem duplicar código ou reinventar a roda. O planejamento gera uma descrição detalhada listando os cenários (caminho feliz, casos de erro), tabelas de banco de dados a serem criadas e os arquivos exatos a serem modificados, garantindo que a IA só faça o que foi pedido. 

Com o planejamento pronto, rodo o comando `execute` na tarefa. Durante a implementação, usamos uma série de agentes (como o agente de banco de dados ou o agente de componentes de front end) operando com suas respectivas "skills". Mantemos também uma pasta "references" com documentações de arquitetura, design system e workflow para guiar os agentes a seguirem as melhores práticas do nosso projeto e evitarem bagunças ou falhas de segurança. Seguir esse processo burocrático e bem arquitetado garante um código organizado, viabilizando a criação de aplicações profissionais fáceis de dar manutenção em produção.
